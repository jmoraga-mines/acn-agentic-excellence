    Background

    Scope Reframe Complete

    Where it started: The project began as an OpenPit simulator with a companion reporting script. The reporting layer was tightly coupled to OP output schemas — it read truck_cycles directly, referenced shovel_id and crusher_feed by name, and had no concept of a second mine type.

    What changed first: The underground simulator (room and pillar, LHDs, hoist, ore pass) was built alongside OP as an independent parallel system. The question of whether minemgr should serve both surfaced the real problem: two simulators would mean two separate reporting codebases with duplicated analysis logic — or one codebase that branches constantly on source type.

    The key reframe: The simulators can be replaced by real operational data. A mine running Modular Mining or Jigsaw exports the same structural data — equipment events, shift summaries, production tables — that the simulators produce synthetically. minemgr is not a simulator companion. It is an operational analysis platform. The data source (OP sim, UG sim, or real mine) is an implementation detail belonging to an adapter.

    Architectural consequence: Adapter/analysis boundary. Adapters translate source-specific schemas (truck_id, shovel_id, crusher_feed) into a source-agnostic common model (equipment_id, origin_id, destination_id). All analysis, reporting, and AI logic operates on the common model. Adding a new data source — a real mine, a different simulator, a manual log import — requires writing one new adapter. No analysis, report, or AI module changes.

    What this means for the product: The analysis IP — variance attribution, plan adherence calculation, hoist OEE analysis, room readiness scoring — is reusable across any compatible data source. The agents (ShiftIQ, Review, Handover) will work against real mine data when real adapters are written, without any changes to the AI layer. The simulator is the development environment; the real mine is the deployment target.
